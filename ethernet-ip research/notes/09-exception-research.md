# 09 - CIP Error Routing and General Status Codes

## Overview

CIP explicitly separates transport success from application-level execution. In EtherNet/IP, the encapsulation layer and Common Packet Format (CPF) may successfully deliver a request to a device, while the CIP Message Router ultimately determines whether the operation can be executed.

When a request references an invalid tag, object path, or unsupported operation, the failure is reported through the **General Status field** within the CIP response payload rather than through the EtherNet/IP encapsulation header.

---

## CIP Routing Failure Structure

A failed explicit message response preserves the encapsulation and CPF structure but modifies the CIP application payload to reflect execution failure at the message router level.

The response structure is as follows:

```text id="c3v8qa"
+-------------------+-------------------+---------------------------------------+
| Field Component   | Data Type         | Purpose                               |
+-------------------+-------------------+---------------------------------------+
| Reply Service     | uint8             | Response service code (0x80 bit set)  |
| Reserved          | uint8             | Alignment padding                     |
| General Status    | uint8             | Primary CIP error indicator           |
| Additional Status | uint8             | Number of 16-bit status words         |
| Extended Status   | uint16[N]         | Optional diagnostic details           |
+-------------------+-------------------+---------------------------------------+
```

The key diagnostic element is the **General Status** field, which identifies why the CIP Message Router rejected the request.

---

## General Status Codes (Observed Behavior)

The following status codes are commonly observed during invalid routing, malformed requests, or unsupported operations in EtherNet/IP environments:

| Status Code | Name                     | Meaning                                                                                           |
| :---------- | :----------------------- | :------------------------------------------------------------------------------------------------ |
| `0x04`      | Path Segment Error       | The request path contains an invalid or unrecognized segment.                                     |
| `0x05`      | Path Destination Unknown | The referenced Class, Instance, or Attribute does not exist.                                      |
| `0x06`      | Partial Data             | The response is incomplete and additional fragments are required.                                 |
| `0x10`      | Device State Conflict    | The controller rejected the request due to its current operating state (e.g., mode restrictions). |
| `0x13`      | Not Enough Data          | The request does not include sufficient parameters for execution.                                 |
| `0x14`      | Attribute Not Supported  | The attribute exists in the object model but is not readable or writable in this context.         |
| `0x26`      | Path Size Invalid        | The encoded request path length does not match the expected format.                               |

These codes are generated by the CIP Message Router and reflect execution-level validation rather than transport-level failures.

---

## Empirical Observation

In tests involving non-existent symbolic tags such as `NON_EXISTENT_TAG`, the controller typically returns:

* Successful EtherNet/IP encapsulation delivery
* Successful CPF parsing
* CIP execution failure at the message router level
* General Status `0x05` (Path Destination Unknown)

This behavior confirms that routing validation occurs after symbolic resolution but before application-level execution.

---

## Security Characteristics

* CIP distinguishes between transport success and application-level failure, allowing precise error signaling at the message router layer.
* General Status codes provide direct insight into object model structure and routing validation behavior.
* Invalid tag or object references can be used to probe device structure without modifying controller state.
* Error responses are deterministic and can be leveraged for fingerprinting differences between implementations.

---

## Key Takeaways

* CIP errors are reported at the application layer using the General Status field, not the encapsulation header.
* A successful EtherNet/IP transfer may still result in a CIP-level failure.
* Status `0x05` indicates an invalid or non-existent object path.
* The Message Router performs validation after CPF parsing but before executing the requested service.
* Error codes are a valuable source of behavioral information for protocol analysis and device fingerprinting.
